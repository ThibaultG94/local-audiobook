## 4) Challenges and Friction Points

1. **Intermittent conversion crashes difficult to reproduce (Story 6.1)**
   - Non-deterministic failures made debugging frustrating
   - Required multiple reproduction attempts with different file types
   - Root cause (mal-normalized package imports) was simple but hard to identify
   - **Resolution:** Full traceback capture with `traceback.format_exc()` made cause evident

2. **Degraded readiness logic complexity (Story 6.2)**
   - Distinction between 'degraded' and 'not_ready' states required careful iteration
   - Edge cases (both engines up, primary missing, empty engines list) needed explicit testing
   - Fallback gating logic needed multiple review cycles
   - **Resolution:** Comprehensive edge case test suite and clear state transition rules

3. **Security vulnerabilities discovered in review (Story 6.3)**
   - Path traversal vulnerability using `str.startswith()` instead of `Path.is_relative_to()`
   - Transaction ordering allowed orphaned metadata
   - Missing confirmation parameter risked accidental deletions
   - **Resolution:** 10 security and data integrity fixes applied during code review

4. **High volume of reference validation (Story 6.5)**
   - 89 markdown links required validation
   - Cross-referencing 6 epics, 30 stories, and planning artifacts was time-intensive
   - Circular reference to Story 6.5 needed correction
   - **Resolution:** Python validation script automated link checking

### Systemic Root Cause

Primary challenge was **comprehensive quality assurance across stabilization work**. Unlike feature development, hardening work requires catching edge cases, security issues, and subtle bugs that only emerge under adversarial review. The team's commitment to rigorous code review was essential to Epic 6's success.

## 5) Deep Story Analysis — Patterns Across Epic 6

### Common Struggles

- **Intermittent failures are debugging nightmares**: Non-deterministic crashes (Story 6.1) required enhanced diagnostic tooling to resolve
- **Security issues hide in plain sight**: Path traversal and transaction ordering bugs (Story 6.3) were only caught through adversarial review
- **Edge cases multiply in stabilization work**: Degraded readiness (Story 6.2) and completion report validation (Story 6.5) required exhaustive scenario coverage

### Recurring Review Feedback

- **Security must be explicit, not assumed**: Use secure APIs (`Path.is_relative_to()`) instead of string operations
- **Transaction ordering matters**: Delete database records before artifacts to prevent orphaned metadata
- **User safety requires confirmation**: Destructive operations need mandatory confirmation parameters
- **Diagnostics must capture full context**: Exception handling should include job_id, correlation_id, and config for root cause analysis

- **Story 6.1 traceback enhancement**: Adding `traceback.format_exc()` transformed opaque crashes into diagnosable failures
