## 4) Challenges and Friction Points

1. **Correlation propagation edge cases required multiple review cycles**
   - Story 5.2 code review identified missing correlation_id fallback generation at worker boundary and import service propagation issues.
   - Required careful boundary-level fallback logic to prevent empty correlation values mid-pipeline.

2. **Event emission coverage gaps discovered late**
   - Missing events for library path validation failures and player load success paths found during Story 5.2 review.
   - Chunk assembly progress events added during Story 5.2 review to improve postprocess observability.

3. **Diagnostics UI state management complexity**
   - Story 5.3 required careful coordination between presenter mapping, view state transitions, and event emissions.
   - Panel clearing on successful conversion completion was missed initially and added during review.

4. **Support workflow category mapping required iteration**
   - Story 5.4 initially missed persistence category test coverage and fallback guidance scenarios.
   - Nested detail sanitization needed recursive implementation to handle complex error structures.

### Systemic Root Cause
Primary challenge was **comprehensive edge-case coverage in distributed instrumentation** across multiple architectural layers. The complexity of maintaining correlation continuity and event consistency across UI/service/adapter boundaries required careful review attention.

## 5) Deep Story Analysis — Patterns Across Epic 5

### Common Struggles
- **Boundary-level correlation propagation**: Ensuring correlation_id remained valid and non-empty across service handoffs required explicit fallback generation at entry points.
- **Event emission completeness**: Achieving symmetric event coverage (both success and failure paths) across all pipeline stages required systematic review.
- **Safe detail exposure**: Balancing diagnostic richness with user safety required recursive sanitization and explicit safe-display checks.

### Recurring Review Feedback
- **Correlation context must never be empty mid-pipeline**: Generate fallbacks only at boundaries, never propagate blank values.
- **Event emissions must be symmetric**: Success paths need events just like failure paths for complete observability.
- **Test coverage must match instrumentation scope**: Every new event emission point needs corresponding test validation.
- **Sanitization must be recursive**: Nested error details require deep sanitization, not just top-level filtering.

### Breakthrough Moments
- **Story 5.1 schema strictness**: Establishing severity whitelisting, event name pattern validation, and extra field size limits created a robust foundation that prevented drift in later stories.
- **Story 5.2 normalized error envelopes**: Standardizing `{code, message, details, retryable}` across all stages enabled consistent diagnostics consumption in UI layers.
- **Story 5.3 safe detail filtering**: Implementing recursive sanitization with explicit unsafe pattern detection protected users from internal implementation details.
- **Story 5.4 category-driven remediation**: Mapping error categories to specific local-only guidance provided actionable next steps without cloud dependencies.

## 9) Significant Discovery Check

### Result
**No fundamental architectural changes required.**

Epic 5 implementation validated the existing architecture's observability design:
- Infrastructure logging layer successfully centralized schema enforcement
- Service/adapter boundaries cleanly separated event emission from logging policy
- UI/presenter layers effectively consumed normalized error contracts
- Non-blocking constraints preserved throughout instrumentation expansion

### Important Technical Discoveries

1. **Correlation propagation requires explicit boundary management**
   - Discovery: Implicit correlation propagation led to empty values mid-pipeline.
   - Learning: Generate correlation_id fallbacks only at entry boundaries (import, worker), never mid-flow.
   - Impact: Established clear pattern for future boundary-crossing instrumentation.

2. **Recursive sanitization is essential for nested error structures**
   - Discovery: Top-level sanitization missed unsafe content in nested error details.
   - Learning: Implement recursive filtering for any user-facing diagnostic content.
   - Impact: Prevents accidental exposure of internal implementation details.

3. **Event emission symmetry improves diagnostic completeness**
   - Discovery: Missing success-path events created incomplete observability.
   - Learning: Every failure event should have a corresponding success event.
   - Impact: Enables complete lifecycle reconstruction from JSONL logs.

4. **Schema strictness prevents downstream drift**
   - Discovery: Strict validation in Story 5.1 caught numerous payload inconsistencies early.
   - Learning: Upfront schema enforcement is more effective than post-hoc validation.
   - Impact: Reduced debugging time in Stories 5.2-5.4.
